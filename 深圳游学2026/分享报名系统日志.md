## 2026-03-31 上线

### 背景
深圳游学约100人，希望安排一个晚上的小组分享活动（4月9日晚9-10点）：
- 先征集愿意分享的话题
- 大家报名感兴趣的话题
- 根据结果选前5个，分5组交流

### 架构决策

| 问题 | 决策 |
|------|------|
| 身份识别 | 手机号（比微信号更准确，不展示给其他用户） |
| 分享者身份 | 提交话题后不能再报名（用同一手机号判断） |
| 报名是否可改 | 可以，同手机号重新提交覆盖 |
| 话题是否可改 | 不可以，确认后提交（避免歧义） |
| 选前几名 | 手动决定（管理员看结果后决定），系统按报名数排序 |
| 数据隔离 | 独立文件 `shenzhen_data.json`，与 Roomie 完全不共享 |
| 部署方式 | 共用 Flask 进程，��由前缀 `/api/shenzhen/`，页面 `/shenzhen` |

### 功能清单

**用户端**
- 提交分享话题：姓名 + 公司 + 职位 + 主题标题 + 介绍 + 手机号
- 浏览话题并报名：单选，可改报名
- 首页实时热度榜：按报名人数排序，每30秒自动刷新
- 话题被删除时提示用户重新报名（localStorage 记录手机号）

**管理端**（`?admin=1`，密码保护）
- 查看所有话题及报名名单
- 删除不合适的话题（软删除，原报名者看到取消提示）

### 时间节点
- 话题提交截止：4月5日 20:00
- 报名截止：4月8日
- 活动时间：4月9日晚 9:00–10:00

### 新增文件
- `shenzhen.html` — 前端页面（部署在服务器 `/shenzhen` 路径）
- `api.py` — 新增 `/api/shenzhen/*` 路由（8个接口）
- 服务器自动创建 `shenzhen_data.json`（首次请求时生成）

### API 接口
| 接口 | 说明 |
|------|------|
| POST `/api/shenzhen/propose` | 提交分享话题 |
| GET `/api/shenzhen/topics` | 获取话题列表（含报名人数） |
| POST `/api/shenzhen/signup` | 报名话题 |
| GET `/api/shenzhen/check_signup?phone=` | 检查是否有被取消的报名 |
| GET `/api/shenzhen/admin/topics` | 管理员查看话题+名单 |

## 2026-04-02 Roomie 主站：用户协议 + 反馈入口

**用户协议弹窗**
- 开始匹配前新增复选框，必须勾选才能提交
- 弹窗内含完整免责声明，「我已阅读，同意协议」按钮自动勾选并关闭

**反馈入口（💬 我有想法）**
- 首页底部新增入口按钮
- 弹窗支持4种类型：💡提建议 / 🐛反馈问题 / 💭许个愿 / 👍表扬一下
- 500字限制，实时字数，公开/私密选项
- 后端：POST `/api/feedback`，存入 `feedback_data.json`

**Bug 修复**
- `submitForm` 里 `const errEl` 被声明了两次，导致整个 script 块解析失败，所有按钮失去响应
- 原因：加协议校验代码时，带入了一行多余的 `const errEl` 声明
- 教训：`const` 在同一作用域内不允许重复声明，会导致整块 JS 静默失效

## 2026-04-04 排名显示调整

**改动**：榜单序号全部改为橙色，不再区分前五名和后面的。

**为什么**：原来前五亮色、后面灰色，会让排在第六、第七的提报者觉得没希望、受挫。话题质量好不好不只看票数，组织者会综合判断。不应该用视觉让人觉得自己"已经出局了"。

---

## 2026-04-04 满员机制上线

**规则**：每个话题报名人数达到 10 人，自动锁定，不再接受新报名。锁定后话题卡片变灰显示「已满」。管理员后台也可手动控制。

**为什么是 10 人**：会场没有麦克风，分成若干小组围坐聊，10 人左右是最舒服的规模。人太多就变成一个人站着讲、其他人听���破坏了破冰的氛围。

**更深的逻辑**：这个活动本来就不是演讲，是破冰。小规模、平等的对话才是目的。满员机制从技术层面保护了这个形式，不让某个热门话题变成"小型讲座"。
